# Paper Note

本 skill 用于将论文（单篇或综述）转化为**领域认知导引级**的深度精读报告，而非简短的会议摘要。
核心目标：提取可迁移的 Insight、暴露设计张力与边界、生成可供讨论的范式级 Q&A。

## 产出物形态选择

| 形态 | 适用对象 | 核心目标 | 信息密度策略 |
|---|---|---|---|
| **单篇研究论文**（Research Paper Mode） | 提出一个具体方法/系统的论文 | 提取该方法的 insight 与边界 | 实验数字极简，只保留"有感觉"的数据；聚焦核心机制与 design trade-offs |
| **大型综述**（Survey Mode） | 50+ 页、引用 100+ 篇的领域全景 survey / taxonomy | 绘制领域地图 + 提出新的认知框架 | **不压缩长度**，详实本身就是价值；里程碑系统替代实验数字 |

## 一、Research Paper Mode（单篇研究论文）

**动笔前提**：必须先锁定 the turn（核心转折/范式翻转）——作者到底推翻了领域里的哪个旧共识？如果找不到这个转折点，报告就会退化成高级摘要，无写作价值。

**结构纪律**：六段结构是启发式骨架，不是强制配额。若某节无 insight 可写（如实验设计平淡无奇），直接删掉，不要为填满格子而硬凑。

### 报告结构

1. **The One-Liner（一句话带走）**
   - 先用一句话概括论文的核心洞察/范式转换。
   - 标准：读完这句话的人，能在 30 秒后向朋友复述作者真正想说什么。
   - 禁止：把标题扩写成一句废话。

2. **Why Now?（背景与痛点）**
   - 不讲领域通识，只讲**作者眼中的特定张力**：当前做法卡在哪个具体环节？所有人都在忍但没解决的痛点是什么？
   - 突出**作者的 unique angle**：为什么这位作者看到了别人没看到的解决路径？
   - 比喻仅用于"区分混淆概念"，禁止装饰性比喻。

3. **The Turn（核心机制）**
   - 不讲流水账（Motivation → Method → Architecture 逐层铺陈）。
   - **拆分设计决策**：作者为什么选择 A 而不是 B？每个关键决策的 trade-off 是什么？
   - 用类比时遵循"看起来像 X，实则是 Y"，目的是切开容易混淆的边界。
   - 保持信息密度，去掉只证明存在性的冗余描述。

4. **Results at a Glance（结果速览）**
   - **极简原则**：只保留"有感觉"的数字——即那些不仅证明效果好，而且能揭示某种深层性质的数据。
   - 用一句话解读每个关键数字，告诉读者"这个数字意味着什么"。
   - 删除所有 baseline 对比表格，除非数字本身 tells a surprising story。

5. **What Remains（余味）**
   - 作者**没有回答**或**刻意回避**的问题。
   - 这个工作**打开的新问题空间**是什么？
   - 用 2–3 句话收束，让读者在合上报告后仍有思考惯性。

6. **Critical Audit（批判性审计）**

   - **定位**：Critical Audit 是批判性思维（critical thinking）在单篇论文精读上的具化执行。它提供纪律化的审查框架，防止发散式批判沦为 trivial 的"我觉得"。
   - 与 What Remains 区分：What Remains 处理作者**已知但未完成**之事（Known Unknowns）；Critical Audit 追问作者**未意识到或未提及**、却对核心结论构成实质性威胁的系统性问题（Unknown Unknowns）。
   - **不写进报告正文的内容**：审计原则、筛选矩阵、方法论层级等元叙述。报告正文只呈现审计结果。
   - **出现原则**：仅在发现符合标准的问题时写入；若遍读全文后未发现，该节删除，禁止为空编造。
   - **有效批判的三条标准**（须同时满足）：
     1. **论文锚点**：指向具体公式、图表、章节或实验配置细节。
     2. **实质性威胁**：能动摇核心结论的适用范围、因果链条或安全声明，而非仅指出「样本量不够」「未来可扩展」等 trivial 疏漏。
     3. **未被提及**：作者未在正文、limitations 或 future work 中以任何方式提及或回避。
   - 审计方向（启发式，非模板）：方法论硬伤（实验设计/指标选择/归因错误）、论证断裂（证据到结论的因果链是否成立）、外推越界（声称的泛化性是否超出证据边界）。

7. **Brainstorm Q&A（发散与辩证）**
   - **不设模板，不设固定问法**。所有问题必须是在精读过程中**由论文内容自然生发**的。
   - 思考方向：<br>质疑假设 / Trade-off 分析 / 边界推演 / 领域对位 / 旁征博引
   - **好问题的信号**（反例："本文创新点是什么？" "未来方向有哪些？"——这类 asking for summary 的模板问题不是思考）：
     - 作者声称 X 能解决问题，但若条件 Y 不成立，X 是否反而制造更大麻烦？
     - Table 3 的 5% 提升来自方法本身还是数据集偏置？
     - 把核心模块替换成更简化的版本，若假设仍 work，说明作者的真正贡献在别处。
   - 数量：3–5 个，完全由论文质量与思考空间决定，不强求。
   - 回答时先给出结论句，再回落到论文内的证据锚点（章节、公式、实验现象）。

## 二、Survey Mode（大型综述论文）

> 综述的使命不是推销一个方法，而是绘制一幅**领域地图**。
> 核心差异：研究论文的核心是"作者挖了多深的井"；综述的核心是"这条河从哪流来、有多少支流、哪里还没被探过"。

### 报告结构（六段式）

**禁止**把综述压缩到单篇论文的 Motivation → Design → Results 三段结构中。

1. **Editorial Thesis（编者论题）**
   - 不是概括"这篇综述说了什么"，而是捕捉**编者如何重新定义了这个领域**。综述的 taxonomy 本身就是一种 argument，不是客观中立的知识图谱。
   - 追问：综述推翻或重构了什么旧认知？它试图让领域从哪种 framing 转向哪种 framing？

2. **Evolution Path（范式演进脉络）—— 核心段落**
   - 取代静态的 taxonomy 骨架，用**时间/因果轴线**串起领域演进。
   - 每个阶段必须回答：**前一范式卡在哪了？新范式是如何回应这个卡点的？**
   - 每阶段只用 1-2 个**标志性系统**做 anchor，附带一句话的"它填补了什么 gap"。其余名字只出现在括号里。
   - 每个出现的缩写/名字，首次出现必须附带 **≤15 字的功能定位**。
   - **禁区**：不要变成"按应用领域逐个扫描的目录"（如 coding → GUI → embodied → science）。

3. **Structural Map（领域结构速览）**
   - 给出板块**因果依赖图**，而不是平铺的目录。
   - 重点说明各章/板块之间的依赖关系（如"没有 §2 对接口属性的定义，§3 的机制层就成了无的放矢"）。
   - 如果综述的章节安排本身就是线性递进，要指出这个递进假设了什么。

4. **Mechanism Deep-Dive（跨阶段关键张力）**
   - 提取 2-3 个贯穿多个阶段的**结构性推力/张力**（如"可执行性 vs 可审计性""静态 harness 配置 vs 自我进化"）。
   - 说明：早期工作如何处理它？为什么失效？后来的工作如何重新 framing？当前走到哪个平衡态？
   - 与研究论文版的区别：单篇论文关注一个具体 design choice 的 trade-off；综述版关注**一个跨时代的张力如何驱动了领域走向**。

5. **Open Problems & Frontiers（开放问题）—— 核心锚点**
   - 把 §5.x 当作综述的**主编判词**，而不是礼貌性的附录。
   - 按**实质性**分级：编者分配了独立章节、给出了可操作子问题、暴露了 current framing 结构性裂缝的 → **高度实质性**；泛泛号召性的 → **礼貌性**。
   - 将 open problem 与综述前文提到的演进脉络挂钩。
   - 如果编者给出了未来方向的可操作性建议（如"需定义 mutation operator、telemetry 标准"），将其保留并突出。

6. **Critical Audit（批判性审计）**

   - **定位**：同 Research Paper Mode——Critical Audit 是批判性思维在单篇论文精读上的具化执行。在综述语境下，审计对象升级为编者的 taxonomy / framing 本身。
   - 在综述语境下，Critical Audit 追问编者的 taxonomy / framing 本身是否存在未被察觉的结构性裂缝。
   - 出现原则与有效标准同 Research Paper Mode：有明确锚点、对核心框架有实质性威胁、编者未提及。
   - **有效批判标准**（同时满足）：论文锚点、实质性威胁、未被提及。
   - 特有的审计方向（启发式）：taxonomy 的 inclusion/exclusion 是否系统性地边缘化了某些研究路径？编者声称的「范式演进必然性」是否有替代解读？某个 Open Problem 若被解决，是否会反过来推翻前面的分类基础？

7. **Brainstorm Q&A（范式辨析）**
   - **数量不封顶**，由文章复杂度决定。
   - **最低保证**：至少涵盖三类维度——(1) 对齐/入门（帮助零先验读者定位核心概念），(2) 张力辨析（编者框架内的深层冲突），(3) 盲区推演（taxonomy 刻意沉默的视角）。
   - **触发条件**：每出现一个新的范式阶段、一个新的核心机制或一个独立的 Open Problem，都应考虑是否值得单独设问。
   - **禁止**：把三类维度误解为"每类只能一问"的定额模板。
   - **综述特有的提问角度——编者议程（editor's agenda）**：
     - 如果全盘接受这个 taxonomy，哪些研究路径会被边缘化？
     - 编者说领域必然从 A 走向 B，是否存在别的解读方式使 A 的失败并不成立？
     - 某个 §5 Open Problem 若被解决，是否会反过来推翻 §2 的分类基础？
   - 回答时先给出结论句，再回落到综述内的证据锚点（章节、编者的显式判断、代表性系统）。

## 三、通用执行约束（适用于所有形态）

| 维度 | 要求 |
|---|---|
| 语言 | 中文为主，保留英文术语与论文原名 |
| 术语注释 | **首次出现的缩写/术语必须附带 ≤15 字的功能定位**。不高估读者先验 |
| 信息密度 | 没感觉的内容直接删；综述形态不允许为压缩篇幅牺牲因果链 |
| 比喻使用 | 仅在"切开混淆概念"或对齐零先验读者时使用；禁止装饰性比喻 |
| 术语表禁令 | 禁止在报告末尾集中挂"术语表"。读者读到中间已卡壳，不会翻到结尾查表；注释必须 inline |
| 跨报告关联 | 单篇报告不预设与其他论文的对比/关联。跨论文联系应在多篇阅读后自然浮现，而非在单篇结尾硬凑 |
| 成功判据 | 读者关上报告后，脑子里留下的应是一个想继续追问的问题，而非一串术语和数字 |
| Open Problems 权重 | **不低于正文段落**。作者/编者暴露的认知边界，往往是全文 insight 最浓处 |
| 时间/篇幅预算 | **无需紧缩**，信息密度优先于篇幅控制。综述尤其如此 |
| Critical Audit | 每篇报告定稿前必须执行 Critical Audit——批判性思维（critical thinking）在单篇论文上的具化审查。评估作者未提及但对核心结论构成实质性威胁的系统性问题；审计结果写入独立段落，若未发现符合标准的问题则删除该段落，禁止为空编造。报告正文不出现审计原则、方法论层级、筛选矩阵等元叙述。 |
| **元话语禁令** | **Metadiscourse Prohibition**：禁止在正文中出现方法论自证语句，如"本章不是…而是…"、"理解 X 比记住 Y 更重要"、"这条线而非静态分类表才是骨架"。方法论选择只能通过内容结构（叙事顺序、因果连接、段落张力）自然体现。**让内容自己证明，不要用元语言宣告**。 |

> **进阶认知规范**：Q&A 设计哲学、Aftertaste 深度写作指南、Open Problem 判别标准等完整方法论资产，详见 `references/reading-playbook.md`。当对某条规则的起源或边界有疑问时，优先查阅该文件。

## 四、PDF 提取与阅读工作流

### 1. 提取策略

- **环境陷阱**：在部分容器环境（如 PAI / Jupyter）中，hermes venv 的 Python 无法直接 `import fitz`（PyMuPDF）。若需使用 PyMuPDF 提取，应显式调用系统 Python：`/usr/bin/python3 <extract_script> <pdf_path>`
- **推荐工作流（使用本 skill 内置脚本）**：
  ```bash
  cd ~/.hermes/skills/paper-note   # 或 skill 实际路径
  /usr/bin/python3 scripts/extract_paper.py <pdf_path> [output_path]
  ```
  该脚本会自动提取元数据（标题、作者、年份、arXiv ID）、跳过参考文献章节、生成带页码标记的文本文件。
- **手动 fallback（无脚本权限或需自定义逻辑）**：
  ```bash
  /usr/bin/python3 -c "
  import fitz, sys
  doc = fitz.open(sys.argv[1])
  text = ''
  for i, page in enumerate(doc):
      text += f'--- Page {i+1} ---\n' + page.get_text() + '\n'
  with open(sys.argv[2], 'w', encoding='utf-8') as f:
      f.write(text)
  " <pdf_path> <output.txt>
  ```
- **无 fitz 时的降级**：可改用 `python3 -m pip install PyMuPDF`（系统 Python 下）或 fallback 到手动提取。
- **超长论文**：提取时应自动跳过参考文献章节以节省处理量。

### 2. 阅读长度策略（关键）

- **多篇论文推荐并行 subagent**：每篇各分配一个独立子任务，同时生成笔记。这是应对多任务的首选策略。
- **单篇长度约束**：若某篇提取文本 ≥ 800 行，其对应的 subagent 极易在生成长响应时超时中断。此时应将**该篇收回主 agent**，通过 `read_file` 分段读取原文后直接撰写；其余短论文仍可并行 subagent 处理。
- **大型综述（Survey）的特有风险**：subagent 对 survey 进行压缩时，极易丢失 paradigm evolution 的因果链和 open problems 的实质性分级。若发现笔记呈"按应用领域逐个罗列的目录感"，必须立即 fallback 到主 agent，重建 Evolution Path。

### 3. 质量 assurance（必读）

- subagent 若基于**压缩/截断/摘要后的输入**生成笔记，极易丢失以下要素：实验配置的精确数值、消融实验的完整粒度、实验命名的完整定义、边界条件与限制。
- 生成后若发现笔记呈"综述感"而非"精读感"，必须立即执行 **Plan A（主 agent 串行精读）**：
  - 主 agent 用 `read_file` 分段读取完整原文（每段 400–600 行）；
  - 同时打开旧笔记对照，逐节标记缺失的精确细节；
  - 重写为包含完整数字、配置、边界条件的修正版报告；
  - 宁可丢弃旧笔记重新精读，也不要保留信息不完整的 summary。

### 4. 输出与清理

- 保存为 `<Paper Name> - Notes.md`，放在 PDF 同目录或约定的 `arxiv/` / `research/` 工作区。
- 删除临时提取文件（如 `paper_note_*.txt`），保持工作区整洁。
