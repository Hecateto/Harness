# AHE: 可观测性驱动的 Coding-Agent Harness 自动进化

> arXiv:2604.25850v4 | Lin et al., Fudan University & Peking University & Shanghai Qiji Zhifeng, May 2026

---

## 1. The One-Liner

Harness 不再靠人手工调，而是让每次编辑都附带可证伪的预测——**从"改了试试看"转向"声明预期→验证→归因"的闭环进化**，可观测性是闭合回路的唯一铰链。

---

## 2. Why Now?

Harness（连接模型与工具/环境的中间层）已被实证为 agent 性能的核心变量，但它的工程方式仍然是手工 craft——改一行 system prompt 可能涨 3 个点也可能跌 5 个点，且无人知道为什么。作者捕捉到的张力：**harness 的每一次编辑都是在黑盒里掷骰子，因为没有结构化的方法把"改了什么"与"结果变了多少"连起来。** 这不是一般的可解释性问题——而是当编辑对象从模型参数变成外部文本配置时，梯度信号消失了，你连"哪个编辑贡献了什么"都无法归因。三根可观测性支柱的本质是重建一条替代梯度链：component observability 给出可编辑单元的边界，experience observability 将原始轨迹压缩为可操作的证据，decision observability 则让每次编辑变成一个可检验的假设。

---

## 3. The Turn

核心 trade-off：**归因精度 vs. 迭代效率**。三根支柱各自做了一次代价选择：

1. **Component observability：文件级表示。** 7 种正交组件类型（system prompt、tool description、tool implementation、middleware、skill、sub-agent configuration、long-term memory）全部暴露为独立文件。好处是编辑粒度清晰、diff 可追溯；代价是文件边界是人为划分的——跨文件的耦合修改（如 tool description 与 implementation 的联动更新）只能靠 experience observability 间接捕获，无法在组件层面声明依赖。消融实验已经暗示了这一点：system prompt 单独编辑反而回归，很可能因为它与其他组件存在强耦合。

2. **Experience observability：分层蒸馏。** 百万级原始 trajectory token → Agent Debugger → 分层 drill-down evidence corpus。这是信息瓶颈所在：蒸馏必然丢信息，保留什么决定了编辑质量。作者选择保留"与失败/成功因果相关"的证据，但因果归因本身依赖下一层——decision observability 的回溯。如果回溯不准，蒸馏就是在噪声上做压缩。

3. **Decision observability：可证伪预测（change manifest）。** 每次编辑必须声明"我改了 X，预期对 Y 类任务有 Z 效果"，下一轮用任务级结果验证。这是全篇最关键的设计：它把 harness 编辑从"盲搜"变成"假设检验"。代价是——manifest 只能在一个 rollout 周期内验证，而一个 rollout 内多种编辑的交叉干扰（concurrent edits）使得单编辑归因变得模糊。算法中的 Attribute 步骤试图拆解，但 fix precision 仅 33.7%（5× 随机）说明拆解远未精确。

4. **Rollback 机制是安全网而非解决方案。** 当 manifest 验证失败时回滚编辑，但 rollback 只防止单步灾难——累积性微小退化（regression precision 仅 11.8%，约 2× 随机）意味着系统几乎无法可靠识别有害编辑，只能靠"整体 pass@1 下降"这一粗粒度信号做兜底。

## 4. Results at a Glance

| 指标 | 数值 |
|------|------|
| Terminal-Bench 2 pass@1 | 69.7% → **77.0%**（10 轮，~32h） |
| vs. Codex / ACE / TF-GRPO | 77.0% > 72.3% > 71.9% > 68.9% |
| SWE-bench-verified 跨 benchmark 迁移 | **75.6%**（最高）+ token 减 12% |
| 跨模型迁移（3 个异族模型） | +5.1 ~ +10.1 pp |
| 自归因 fix precision | 33.7%（5× random） |
| 自归因 regression precision | 11.8%（~2× random） |

跨 benchmark 迁移同时砍掉 12% token 是最有说服力的数字——说明 harness 进化不是在"记忆"特定任务，而是学到了更高效的操作模式。但 regression precision 11.8% 令人不安：系统几乎看不清自己在哪里在搞破坏。

---

## 5. What Remains

- **Regression 检测为何如此弱？** 11.8% 精度意味着系统对有害编辑近乎盲区。是因为 manifest 的预测粒度太粗，还是因为有害编辑的效果被其他正向编辑的噪声掩盖？如果无法可靠识别退化，闭环就缺了负反馈半边。
- **System prompt 单独编辑回归的根因未深挖。** 消融显示 tools、middleware、memory 各自独立贡献提升，唯独 system prompt 反向。这说明 system prompt 的作用可能完全依赖与其他组件的协同——但论文未进一步拆解协同模式。
- **10 轮迭代后是否已收敛？** 69.7% → 77.0% 的曲线形状未展示。如果还在上升，32 小时的成本是否意味着更长的进化有更大收益？如果已饱和，瓶颈在哪里——是 experience observability 的蒸馏上限，还是 decision observability 的归因天花板？
- **7 种组件类型是否穷尽了 harness 的可编辑空间？** 未讨论组件类型的发现机制——如果存在第 8 种关键组件类型，当前框架能否自动识别并暴露它？
- **跨模型迁移的下界未刻画。** +5.1 到 +10.1 pp 的跨度很大，什么模型特征决定了迁移增益的大小？

---

## 6. Critical Audit

**Self-attribution 的低精度可能系统性偏袒"保留有利编辑"而非"拒绝有害编辑"，导致进化过程单向漂移。** Fix precision 33.7% 意味着被标记为"有效修复"的编辑中 2/3 是误归因；regression precision 11.8% 意味着有害编辑中近 9/10 未被识别。两者叠加，系统倾向于保留过多编辑（因为"看起来有用"）而很少回滚（因为"看不出有害"）。论文用整体 pass@1 的提升来证明系统有效，但未分析这 7.3pp 的提升中有多少来自真正的因果改进、多少来自随机正向波动的累积。若 rollout 的任务采样方差较大，部分提升可能仅仅是选择性保留正向噪声的结果。论文未提供 shuffle 对照或 bootstrap 置信区间来排除这一可能性。

## 7. Brainstorm Q&A

**Q1: decision observability 的 manifest 能否从"二值验证"升级为"连续梯度估计"？**
→ 可以但代价高昂。当前 manifest 本质是"改 X → 任务 Y 是否通过"，是离散信号。如果要估计连续影响（改 X → 任务 Y 的通过概率变化 Δp），需要每个 edit variant 跑多次 rollout，成本指数增长。折中方案：用 Bayesian bandit 在少量 rollout 中估计 Δp 的后验，但这又引入了先验选择问题。→ 锚点：§3.3 change manifest 与 Attribute 步骤

**Q2: 如果把 harness 进化视作一种"外部 RL"，那 policy gradient 的等价物是什么？**
→ 是 experience observability 的蒸馏结果 + decision observability 的归因信号。distilled evidence 相当于 reward signal 的压缩表示，manifest attribution 相当于 credit assignment。但与标准 RL 不同，这里的"参数"是离散的文本片段，更新是替换而非微调——更接近 evolution strategy 而非 policy gradient。→ 锚点：Algorithm 1 Rollout→Clean→Attribute→Distill→Edit→Commit

**Q3: system prompt 为何单独编辑回归？是否暗示 prompt 的作用机制是"约束其他组件的行为空间"而非"直接提供知识"？**
→ 极可能是。System prompt 改了但 tools/middleware 未同步调整，相当于只改了说明书没改工具——agent 行为与新指令不一致，反而更差。这也解释了为什么 tools、middleware、memory 单独改有效：它们直接改变可执行能力，对行为的影响不依赖其他组件配合。→ 锚点：§5 component ablation 结果

**Q4: 跨 benchmark 迁移同时减 token 12%，说明 harness 学到了什么？**
→ 更精简的操作路径。进化可能淘汰了冗余的 tool 调用链、减少了不必要的子 agent 派发、或让 middleware 在更早阶段过滤掉无效分支。这比单纯提升 pass@1 更有价值——它暗示 harness 进化在优化"策略效率"而非"策略覆盖"。→ 锚点：§4.3 SWE-bench-verified 迁移结果

**Q5: 如果 regression precision 无法提升，是否应该放弃闭环归因而转向纯开环的 diversity-driven 进化？**
→ 不一定放弃闭环，但可以考虑不对称策略：对正向编辑保持低门槛接纳（当前做法），对回滚则用更保守的集体判据——如连续两轮整体下降才回滚整组编辑，而非依赖单编辑归因。这牺牲了精细归因但提高了 regression 检测的鲁棒性。→ 锚点：§5 self-attribution 结果，rollback 机制
