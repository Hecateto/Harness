# AutoHarness: improving LLM agents by automatically synthesizing a code harness

> **arXiv: 2603.03329v1** | Google DeepMind

---

## 一句话带走

与其让更大的 LLM 来避免犯错，不如让一个小模型**自己写一段 Python 代码给自己当缰绳**——这段代码能自动学会什么是合法动作，最终让小模型在 145 个游戏中零非法动作，且对战表现超过大模型。

---

## 背景与痛点（Why Now?）

当前 LLM agent 的致命裂缝不在"聪明不聪明"，而在**懂规则不等于守规则**。

Kaggle GameArena 国际象棋竞赛里，Gemini-2.5-Flash 78% 的败局不是因为策略失误，而是因为走了**非法棋步**。这暴露了一个深层悖论：模型在文本层面"理解"了规则（能背诵马走日、象走田），但在逐回合决策中却屡屡越界——就像一个人背熟了交规，上路时仍然闯红灯。

传统解法两头不讨好：
- **微调 LLM**——慢、贵、且会冲淡通用能力；
- **手写 harness**——每个新环境都要重新搭脚手架，无法规模扩展。

作者的 unique angle：既然 LLM 已经能写代码，何不让它**为自己的弱点写补丁**？不是去微调大脑，而是给大脑配一副自动生成的"机械外骨骼"。

---

## 核心机制（The Turn）

论文把 harness 的生成形式化为**程序空间上的树搜索**，LLM 充当突变算子（mutation operator——在候选代码树上执行结构/逻辑变异的探索算子），环境反馈充当裁判。

### 三种 harness 形态（从松到紧）

| 形态 | 机制 | LLM 在推理时的角色 |
|---|---|---|
| **Action-Verifier**（主力） | LLM 提议动作 → `is_legal_action()` 检验合法性 → 非法则重试 | 仍然参与决策，但被代码护栏拦截 |
| **Action-Filter** | `propose_action()` 生成合法候选集 → LLM 负责排序/选择 | LLM 只做策略选择，不做合法性判断 |
| **Policy**（极限） | 代码直接输出动作，推理时**零 LLM 调用** | 完全被代码替代 |

### 搜索策略：Thompson Sampling 指导代码进化

不是简单 iterative prompting，而是维护一个**代码假设树**。每个节点代表一段候选代码， heuristic 值为合法动作准确率。Thompson sampling 平衡探索（尝试全新逻辑结构）与利用（打磨局部可行的代码）。

关键设计细节：
- **双层精炼策略**：若 `is_legal_action()` 误判（返回 True 但动作非法），则同时 refine `propose_action` 和 `is_legal_action`；若正确拦截（返回 False 且确实非法），则只 refine `propose_action`。
- **Reward 融入**：Policy 模式下 heuristic 设为 `H = 0`（非法）或 `H = 0.5 + 0.5r`（合法），把稀疏的环境 reward 注入搜索信号。

### 为什么是"代码"而不是"提示词"？

提示词层面的规则（"请不要走非法棋步"）是**软约束**，模型可以选择性忽略；而代码 harness 是**硬约束**，非法动作在到达环境之前就被物理拦截。更关键的是，代码可以被**执行、测试、版本化、复用**——这是自然语言规则无法做到的。

## Brainstorm Q&A

### Q1: 树搜索优化的是代码，但最终战胜了更大的模型。这本质是"用结构补能力"还是"用搜索补数据"？

这是一个重要的定位问题。如果把它理解成"用搜索补数据"（类似 AlphaCode 的大规模采样），那么 scaling 的终点仍然是更大的模型；但如果理解成"用结构补能力"（给模型一个可执行的中间表示），那它的潜力在于**把 LLM 从"决策者"降级为"代码生成器"**，而真正的决策交给编译后的程序。

论文的 Harness-as-Policy 结果强烈支持后者：当代码完全替代 LLM 时，平均奖励 0.870 不仅超过 Gemini-2.5-Pro（0.707），还超过 GPT-5.2-High（0.844），且推理成本近乎为零。这暗示在某些规则明确的领域，**符号化的代码策略可能比神经化的 LLM 推理更可靠、更便宜**。

但边界也很清晰：作者明确说 2P 游戏的复杂策略需要 MCTS（Monte Carlo Tree Search，基于随机模拟评估走子价值的搜索算法）类方法，而代码生成这类策略"具有挑战性"。所以"代码替代 LLM"目前只适用于规则空间可穷举/可验证的任务。

### Q2: Thompson sampling 的 heuristic weight 固定为 1.0，这个选择是否过于乐观？

论文设定 heuristic weight = 1.0，意味着对历史表现完全信任。在程序合成这种**稀疏奖励、非平稳**的空间里，这是一个强假设——一段代码在 game A 上表现好，不代表在 game B 上也适用。

虽然论文报告了平均 14.5 次迭代收敛，但隐藏了一个有趣的异常值：GermanWhist-v0、Cryptarithm-v0、Othello-v0 和 Chess-v0 需要远多于平均的 LLM 调用。这恰恰说明在**规则耦合复杂、边界条件微妙**的任务上，简单的平均准确率 heuristic 会陷入困境。如果引入类似 UCB（Upper Confidence Bound，平衡探索与利用的置信上界准则）的上界置信或对代码结构相似度的先验，是否会降低这些异常值的开销？

### Q3: 145 个游戏各自生成独立 harness，这能扩展到真实世界吗？

论文坦承当前局限：每个环境独立训练一个 harness。但真实世界的 agent 不会只玩一个游戏——今天的用户要求 agent 同时处理代码、网页、文件系统。

未来的破局点可能是**可复用 harness 库**：识别跨任务的通用约束模式（如"不要覆盖已有文件"、"不要访问未授权 URL"），把它们沉淀为可组合的代码模块。但这需要一个统一的"动作合法性"描述语言——而当前每个游戏的环境接口各不相同，迁移成本极高。

## 结果速览

| 指标 | 数字 | 一句话解读 |
|---|---|---|
| 非法动作导致失败占比 | 78% (Kaggle Chess) | 问题不是"不聪明"，而是"不守规矩" |
| 145 游戏合法动作成功率 | **100%** | 小模型自生成的护栏能完全消除违规 |
| 2P 游戏胜率 (vs Pro) | **56.3% vs 38.2%** | 带护栏的小模型 > 裸奔的大模型 |
| 1P 游戏平均奖励 | **0.745** (vs Pro 0.707) | 护栏不仅防错，还提升了策略水平 |
| Harness-as-Policy 平均奖励 | **0.870** (vs GPT-5.2-High 0.844) | 纯代码策略可以碾压顶级 LLM，且成本趋零 |
| 平均收敛迭代数 | 14.5 次 | 通常不到 20 次 LLM 调用即可生成可用 harness |

## 余味

作者展望了一个递归自进化的图景：把各个领域的 harness 专家蒸馏回基础模型，让模型越用越会写 harness。但一个悬而未决的问题是——**如果 LLM 本身在生成 harness 时犯了错（比如生成的 `is_legal_action` 有 bug），谁来给 harness 套 harness？**

这种"元验证"的缺失意味着 AutoHarness 目前只适用于**有明确、廉价、可自动判定的环境反馈**的场景（游戏、编程、形式化系统）。在反馈模糊、延迟长、代价高的真实世界（商业决策、医疗诊断、社会交互）， harness 的收敛是否会失效？

更深层的问题是，145 个游戏各自独立生成 harness，这种**单环境单 harness** 的模式与真实世界中 agent 需要同时操作多源异构系统（代码库 + 网页 + 数据库 + API）的需求存在根本张力。跨任务的通用约束模式（如"不要覆盖已有文件"、"不要访问未授权 URL"）能否被沉淀为可组合的代码模块，取决于是否存在统一的"动作合法性"描述语言——而当前每个游戏的环境接口各不相同，迁移成本极高。这也呼应了 CodeAgentHarness 综述中提出的 open problem：transactional shared program state 的缺失，本质上就是跨任务 harness 缺乏统一语义基底的症状。

最后，Harness-as-Policy 在 1P/2P 游戏中表现出的压倒性优势（0.870 vs GPT-5.2-High 0.844）提出了一个反直觉的判据：**在规则可穷举的领域，符号化代码策略可能比神经化推理更优**。但这个判据的适用范围恰好与 LLM 的擅长领域（开放域推理、常识推断、 creative generation）形成互补而非重叠。未来 agent 系统的 likely 形态可能不是"LLM 替代一切"或"代码替代 LLM"的二选一，而是一个动态分层的 runtime：LLM 负责高层 intent 理解和异常处理，代码 harness 负责规则明确子空间的精确执行，二者之间的边界由 harness 自身的进化重新定义。
