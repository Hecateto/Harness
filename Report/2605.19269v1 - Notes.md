# CODA: Rewriting Transformer Blocks as GEMM-Epilogue Programs

## 1. The One-Liner

把散落在 GEMM（通用矩阵乘法）之外的 memory-bound 算子"塞回"GEMM 输出 tile 还在片上时就做完——用 epilogue 编程替代独立 kernel 调度，消解 Transformer 训练中最顽固的内存瓶颈。

## 2. Why Now?

Transformer 训练的算力版图极不均衡：GEMM 吞掉绝大部分 FLOPs，却被数十个轻量但 memory-bound 的零碎算子（RMSNorm、SwiGLU、RoPE、残差加法、cross-entropy）反复打断流水线，每次打断意味着一次 HBM 往返。Hopper 架构的 TensorCore accumulator layout 恰好让相邻 feature 对天然对齐在 accumulator 寄存器中——这让 pairwise 操作（SwiGLU、RoPE）首次可以在 epilogue 阶段零额外搬运完成。硬件变了，但软件抽象还停在"一个算子一个 kernel"的旧范式，这个 gap 就是 CODA 的切入点。

## 3. The Turn

**核心 trade-off：牺牲算子通用性，换取片上计算的零拷贝执行。**

CODA 的设计决策可以拆成三层：

1. **保持 GEMM mainloop 不动，只开放 epilogue 扩展点。** 这不是"在 GEMM 后面挂钩子"那么简单——它意味着所有被融合的算子必须能被重参数化为 tile-local 计算。不能 tile-local 化的（如 RMSNorm 的全行归约），就拆成"partial reduction 在 epilogue + auxiliary kernel 收尾"两段。代价是编程模型变窄：只有符合 tile 语义的算子能被原生融合，其余只能走 fallback。

2. **RMSNorm 的代数重参数化是全篇最关键的 trick。** 经典路径是 GEMM→写回 HBM→Residual Add→RMSNorm→写回→下一 GEMM。CODA 利用了 $r$（RMSNorm 的 row-wise scale）与 $W_1$ 的可交换性：先在第一个 GEMM 的 epilogue 做 partial sum-of-squares（tile-local），auxiliary kernel 完成全行归约算出 $r$，然后在第二个 GEMM 的 epilogue 用 $r$ 做行缩放。这把两个 GEMM 之间的三次 HBM 写回压到一次。代价是引入一个极轻量的 auxiliary kernel，且要求 $r$ 与 $W_1$ 可交换——这个条件并非对所有归约操作都成立。

3. **Theorem 1（forward tile-local ⇒ backward tile-local）是可组合性的根基。** 它保证了：只要前向的 epilogue 是 tile-local 的，反向传播自动也是 tile-local 的，不需要逐算子手写反向。这大幅降低了融合算子的开发成本，但也把适用范围锁死在 tile-local 这个条件内——跨 tile 依赖的反向传播（如某些通信密集的并行模式）无法被自动覆盖。

4. **LLM-oriented authoring 是工程上的务实选择。** 将 epilogue 的编程空间约束在专家级 GEMM mainloop 周围，让 LLM 在这个受限空间内搜索组合，而非从零生成 kernel。这绕开了 LLM 写 CUDA 的可靠性问题，但同时也意味着发现新的非平凡重参数化仍依赖人类洞察。

## 4. Results at a Glance

| 场景 | 加速比 |

| GEMM-Residual-RMSNorm-GEMM | 最高 1.2×（人工） |
| RMSNorm / SwiGLU backward | **1.6–1.8×** |
| 整层 forward+backward | 最高 1.2× |
| Cross-entropy 作为 GEMM epilogue | 最高 1.1× |
| 数值精度 | CODA 重参数化路径误差 **低于** 标准 PyTorch |

backward 1.6–1.8× 是最有感觉的数字——说明瓶颈确实在被击碎，而非仅是 margin 改善。数值误差反而更低则出乎预期：减少中间舍入环节的副作用成了正收益。

## 5. What Remains

- **多卡并行下的融合边界未讨论。** TP/PP/EP 场景中，通信原语（AllReduce、AllGather）是否也能被吸收进 epilogue？论文只展示单卡结果。
- **dynamic shape / 变长序列的支持情况不明。** epilogue 的 tile-local 语义依赖固定的 tile 划分，padding 和 variable-length 对 occupancy 和数值的影响未提及。
- **auxiliary kernel 的调度开销是否可被完全隐藏？** 对极小 batch 或极短序列，partial reduction → auxiliary kernel → 第二 GEMM 的三段流水是否仍然占优？
- **可交换性条件的适用边界。** 论文只证明了 $r$ 与 $W_1$ 可交换，但 LayerNorm 的 bias 项、其他归约形式是否也能被类似拆解？未做系统性讨论。
- **LLM-authored kernel 的搜索成本和覆盖率。** 仅展示了"可以达到人工水平"，但没给出搜索所需 token 数、失败率、以及哪些 primitive 组合是 LLM 系统性找不到的。

## 6. Critical Audit

**RMSNorm 重参数化的可交换性假设在低精度下可能失效。** 论文利用 $r(Wx) = W(rx)$ 来延迟行缩放至第二个 GEMM epilogue，但该等式在 FP8/BF16 有限精度下并非严格成立——$W$ 的行内元素可能因缩放顺序不同而触发不同的舍入路径。论文报告了数值误差更低，但测试仅对比 PyTorch 默认路径，未对比"先缩放再乘 $W$"与"先乘 $W$ 再缩放"在 FP8 下的逐元素差异。若 FP8 训练成为主流，这个舍入不对称性可能成为精度隐患。论文未提及此风险。

## 7. Brainstorm Q&A

**Q1: epilogue 融合的终极边界在哪里？——哪些算子永远无法被 tile-local 化？**
任何需要全局归约后再广播到每个 tile 的操作（如 Softmax 的分母、BatchNorm 的统计量）本质上需要跨 tile 通信，只能走 partial reduction + auxiliary 两段式。CODA 对这类算子的处理能力取决于 auxiliary kernel 的延迟是否能被下一个 GEMM 隐藏，而非融合本身。→ 锚点：§3.2 RMSNorm 拆分策略

**Q2: 如果 FlashAttention 也被视为一种 GEMM-epilogue 模式，CODA 能否统一它？**
FlashAttention 的 tiling 策略是对 Q 的行和 K 的列做分块累加，其 epilogue 是 online softmax 归约——这恰好是 CODA 的 tile reduction primitive 的超集。但 FlashAttention 的 mainloop 不是标准 GEMM（有 rescaling 步骤），所以"保持 GEMM mainloop 固定"这个约束会成为障碍。→ 锚点：§2.2 五类 epilogue primitive，§4 Theorem 1

**Q3: Theorem 1 能否扩展到跨 tile 依赖的反向传播？**
定理证明依赖 tile-local epilogue 的链式分解。如果前向引入跨 tile 通信（如 auxiliary kernel 的结果被多个 tile 共享），反向传播的 tile-local 性就不再自动保证——需要显式处理梯度聚合。这意味着 CODA 的"自动反向"优势在更复杂的融合模式下会衰减。→ 锚点：§4 Theorem 1 证明

**Q4: LLM-authored 和 human-authored 的性能差距来源是什么？**
1.2× vs 1.1× 的差距可能源于 LLM 在 epilogue primitive 的组合顺序上做了更保守的选择（例如更少的寄存器复用），而非算法层面的差异。如果差距仅是调参问题，那么 scaling LLM 的搜索预算就能弥合；如果是 LLM 无法发现某些非平凡重参数化，则人类洞察仍然是不可替代的。→ 锚点：§5.3 LLM authoring 实验

**Q5: CuTeDSL 的约束是否在保护 CODA，也在限制 CODA？**
CuTeDSL 提供了 tile 语义的类型安全，让 epilogue 编程不至于写出越界访问。但这也意味着任何需要突破 tile 边界的优化（如跨 tile 的寄存器共享、自定义 shared memory 布局）都难以在当前框架内表达。CODA 的性能上限可能被 CuTeDSL 的抽象天花板锁住。→ 锚点：§3.1 基于 CuTeDSL 的实现
