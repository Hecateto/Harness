# 从建议到护栏：让 LLM Agent 的技能可执行、可验证、可演化

> arXiv:2605.17734v1 | Liu, Ming, Joty, Zhao (NYU, Salesforce AI Research), May 2026

---

## 1. The One-Liner

HASP 把 agent 技能从"文本建议"升级为"可执行护栏"——技能不再是模型可能忽略的 prompt，而是在失败易发状态自动激活、修改动作或注入修正上下文的程序函数，从而让技能的执行力从"建议级"跃迁到"合约级"。

---

## 2. Why Now?

当前 agent 技能编码的主流范式是文本指导（textual guidance）：把经验写成自然语言提示塞进 prompt。问题在于 LLM 对 prompt 的遵从是概率性的——同样的技能描述，模型有时照做有时无视，而失败恰恰发生在模型最需要指导的时刻。这构成了一个结构性矛盾：**技能越需要被遵守的时候，越可能被忽略**。HASP 的 unique angle 是：不做更强的 prompt，而是把技能的激活条件（should_activate）和干预逻辑（intervene）写成可执行代码，使技能的执行不再依赖模型的自觉性，而依赖程序的条件判断。

---

## 3. The Turn

### 设计决策 1：PF（Program Function，带激活条件与干预接口的可执行技能单元）的双模干预

Action override（动作覆盖，直接修改下一步动作）vs. Context injection（上下文注入，添加修正性上下文）。Trade-off：前者强制力高但灵活性低——覆盖了模型可能自行修正的动作；后者保留模型自主性但依赖模型理解修正意图。HASP 让两者共存而非二选一，本质是承认不同失败模式需要不同力度的介入：行为级错误（如重复搜索同一 query）需要直接覆盖，认知级偏差（如忽略矛盾证据）需要上下文注入。

### 设计决策 2：四信号聚合用于 post-training

Timing（激活时机）、Mode（干预模式）、Correctness（干预是否正确）、Outcome（最终结果）四个信号被聚合为训练信号。关键 trade-off：Correctness 信号需要 ground truth 标注干预本身的对错，而 Outcome 信号只需要最终结果——前者更精确但标注成本高，后者更粗粒度但可自动获取。三种训练变体（SFT / RS / OPD）实质是在精确度-自动化轴上的三个刻度。

### 设计决策 3：自演化的过滤机制

Unfiltered evolution 在 web-search 上从 60.3% 暴跌至 36.3%——未经校验的 PF 候选比没有 PF 更糟。HASP 用 executable validation（在回放轨迹上运行候选 PF 验证效果）+ teacher review 做双重过滤。这揭示了一个深层 trade-off：自演化的收益完全取决于过滤的可靠性，而过滤本身依赖已有 PF 的覆盖度和 teacher 的判断——系统起步时的冷启动问题被过滤机制放大了。

## 4. Results at a Glance

|---|---|---|---|
| Baseline (Search-R1) | 29.9% | — | — |
| PF-only | 51.0% | 35.9% | 63.4% |

- PF-only vs. RA-Agent：51.0% vs. 31.2%（+19.8pp），可执行护栏本身的价值就已显著
- Unfiltered evolution：36.3%（比 PF-only 还低 14.7pp），无过滤的演化是负收益
- 内化差异：行为纠正类 PF（如 multi_hop_reasoning_failure）100% silent（模型已学会，PF 不再激活）；输入依赖类 PF（如 decompose_complex_question）仅 3-12% silent——纠正行为容易，补全认知能力难

## 5. What Remains

1. **PF 的组合爆炸**：论文展示的是单 PF 或少量 PF 的效果，当 PF 库规模增长时，多个 PF 同时 should_activate 的冲突如何解决？优先级？合并？论文未讨论。
2. **跨域迁移**：所有实验在同一骨干（Qwen2.5-7B-Instruct）上完成。PF 是否可跨模型迁移？一个为 7B 模型设计的 PF 在 72B 模型上是否会过度干预？
3. **沉默率的两极化**：行为纠正类 PF 100% silent 说明模型已内化该技能，但输入依赖类 PF 几乎不 silent——后者意味着 PF 持续占用推理开销但始终无法被内化。这暗示 PF 对某些失败模式的定位是"外挂"而非"教会"，但论文未分析何时该放弃 PF 转而修改模型本身。
4. **Teacher review 的瓶颈**：自演化的双重过滤中，teacher review（用强模型审核候选 PF）是不可自动化的环节。随着 PF 库增长，teacher review 是否成为扩展瓶颈？论文未讨论规模化路径。

## 6. Critical Audit

### 可执行护栏的可信度假设

PF 的 should_activate() 判断失败模式是否出现，但这个判断本身是代码逻辑——它对失败模式的识别是硬编码的模式匹配，而非对模型状态的真正理解。当模型以 PF 作者未预见的方式失败时，should_activate 可能不触发（假阴性）或误触发（假阳性）。论文报告了 PF 的整体效果，但未拆解 should_activate 的准确率——这是护栏系统的核心可靠性指标。假阴性意味着护栏在需要时缺席，假阳性意味着护栏在不需要时干扰。论文中 unfiltered evolution 的灾难性结果（36.3%）暗示：错误 PF 的危害远大于无 PF 的 baseline，这反过来对 should_activate 的精确度提出了极高要求，而论文未提供该精确度的直接测量。

## 7. Brainstorm Q&A

**Q1: PF 和传统软件中的 aspect-oriented programming（AOP，横切关注点编程）是什么关系？**
→ PF 本质上是 agent 行为层面的 AOP：should_activate 是 pointcut（切点，定义拦截位置），intervene 是 advice（增强逻辑，定义拦截后的动作）。但 AOP 的 advice 是确定执行的，PF 的 intervene 仍然可能被模型忽略（Context injection 模式下）。锚点：PF 的双模干预设计，Action override 更接近 AOP，Context injection 更接近装饰器。

**Q2: 为什么行为纠正类 PF 能 100% 内化，而输入依赖类 PF 几乎不能？**
→ 行为纠正（如"不要重复搜索"）是负约束——模型只需要停止做某事，这可通过 SFT 轻松学会。输入依赖（如"分解复杂问题"）是正能力——模型需要学会何时及如何做某事，这要求模型获得新的认知策略而非仅仅抑制旧习惯。负约束是减法，正能力是加法，后者的学习难度天然更高。锚点：decompose_complex_question 的 3-12% silent rate。

**Q3: HASP 的自演化与 Voyager 的技能库演化有何本质区别？**
→ Voyager 的技能是能力扩展——新技能让 agent 能做之前做不到的事；HASP 的 PF 是故障修复——新 PF 让 agent 不做之前做错的事。前者是探索驱动的，后者是修复驱动的。这导致演化筛选标准不同：Voyander 验证技能是否可执行成功，HASP 验证 PF 是否减少失败——后者需要对比有/无 PF 的轨迹差异，筛选成本更高。锚点：executable validation 在回放轨迹上运行候选 PF。

**Q4: 如果 PF 库足够大，should_activate 的多重触发是否会导致干预冲突？**
→ 几乎必然。两个 PF 可能分别建议 Action override 和 Context injection，甚至两个 Action override 指向不同动作。这映射到操作系统中的信号处理问题：多个信号同时到达时的优先级调度。HASP 的模块化设计回避了这个问题，但规模化时必须面对。锚点：论文未讨论多 PF 并发激活的场景。

**Q5: PF 的"可执行"属性是否也是其脆弱性来源？**
→ 是。文本指导的错误最多是被忽略，PF 的错误会被强制执行。一个 should_activate 误判的 PF 会在模型本应自主行动时强制覆盖其动作——这是护栏从保护变为束缚的时刻。可执行性的"不可忽视"是双刃剑：正是它让 PF 比文本指导更有效，也正是不容忽视让它比文本指导更危险。锚点：unfiltered evolution 的 36.3% 即为证据。
