# Harnessing LLM Agents with Skill Programs

> **arXiv: 2605.17734v1** | NYU / Salesforce AI Research

---

## 一句话带走

与其把经验写成"建议"塞进 prompt 祈祷模型照做，不如把它编译成一段**带触发器和执行体的代码函数**——在模型即将犯错的那一步，直接接管或纠正它的下一个动作。

---

## 背景与痛点（Why Now?）

LLM agent 的失败并非随机，而是**高度结构化地重复**：过早 finalize、重复无效搜索、实体混淆、证据不足就下结论。现有解法是把失败经验提炼为"技能文本"注入 prompt（"请先验证证据再下结论"），但这套机制有个致命软肋——

**它是建议，不是命令。** 模型可以"听到"技能，然后在 reasoning 中背诵一遍，最后依然我行我素。文本技能无法精确指定"何时激活"、"如何修改动作"，更无法保证执行。

HASP 的 unique angle 是**控制反转（inversion of control）**：不再让模型读取技能后自主决定遵不遵守，而是在模型即将执行动作的前一刻，由外部 harness 判断当前状态是否命中某个失败模式——如果命中，就直接重写动作或注入纠正上下文。技能从"教科书"变成了"紧急制动系统"。

---

## 核心机制（The Turn）

HASP 的核心抽象是 **Program Function (PF)**，一个可执行的状态–动作干预函数，标准化接口如下：

- `should_activate(state, proposed_action) → bool`：当前是否处于该技能应介入的状态？
- `intervene(state, proposed_action) → (revised_action, context)`：如果激活，如何修复？

### 两种干预姿势

| 类型 | 机制 | 例子 |
|---|---|---|
| **Action Override** | 直接替换下一个可执行动作 | 把 over-constrained 的 search query 重写为更简洁的检索词 |
| **Context Injection** | 向 reasoning trace 注入纠正性上下文 | 在模型即将 finalize 前注入 "[DECOMPOSITION HINT] 请先拆分为子问题" |

机制分析显示，65.1% 的 PF 触发属于 action override，34.9% 属于 context injection。这意味着 HASP 大部分时间不是在"提醒"模型，而是在**物理层面修改它的下一步行动**。

### 三条互补的使用路径

1. **Inference-time Harness**：PF 作为外部控制层包裹 base agent，零训练即可获得显著提升。Web-search 上从 multi-loop ReAct（Reasoning + Acting，交替推理与工具调用的 agent 范式）的 31.2% 提升到 PF-only 的 51.0%，再加 teacher 辅助选择后达到 56.2%。
2. **Post-training Internalization**：PF 每次干预都留下结构化记录 `(state, original_action, corrected_action, mode, outcome)`，据此生成四元信号 `(timing, mode, correctness, outcome)` 用于 SFT（Supervised Fine-Tuning，有监督微调）、Rejection Sampling 或 On-policy Distillation。
3. **Controlled Evolution**：用当前 checkpoint 重新审视失败案例，从 recurring failure-repair 模式中提炼候选 PF，经**可执行验证 + teacher review** 双门过滤后才纳入技能库。

### 为什么是"代码"而非"提示词"的又一次变奏？

没有执行力的技能只是备忘录。PF 的硬性接口（必须能 mock execute、必须返回合法类型）确保每条技能都是**可验证的行为契约**，而不是模糊的文本建议。

## Brainstorm Q&A

### Q1: 文本技能与 PF 的本质边界到底在哪？

表面上看，两者都是"把经验编码进 agent 的循环"。但它们的根本差异在于**控制权的归属**：

- **文本技能**：控制权在模型手中。prompt 里写"不要过早 finalize"，模型读完后依然可以选择忽略——因为它拥有 action generation 的完整权限。
- **PF**：控制权在 harness 手中。模型生成 proposed_action 后，PF 的 `should_activate` 和 `intervene` 在 action 被执行前进行**强制审查与重写**。模型只能拿到 revised_action。

这类似于操作系统里的 **user space vs kernel space**：文本技能是用户态的建议，PF 是内核态的钩子。

Table 5 的 ablation 为这一定位提供了强证据：去掉 PF、仅保留 teacher 时，web-search 从 56.2% 降到 50.7%（-5.5）；而仅保留 PF、去掉 teacher 时，只降到 51.0%（-5.2）。PF 本身贡献了大部分增益，teacher 只是精修。更关键的是，Prompt-Only Skills 只有 20.5%，说明**文本形式的经验传递效率极低**。

### Q2: PF 的 Post-training 到底 internalize 了什么？

一个有趣的发现是：post-training 后，behavior-correcting PFs（如 `multi_hop_reasoning_failure`、`retrieval_failure`）在 100% 的先前触发案例上变得"沉默"，而 input-dependent PFs（如 `decompose_complex_question`）只有 3–12% 沉默。

这意味着模型学到的是**通用行为模式**（多搜索、少 finalize、写短查询），而不是具体问题的解法。这符合一个直觉：你可以教会一个学生"先验算再交卷"，但没法让他记住每一道题的验算步骤——后者必须靠考试时（inference-time）的实时提示。

但这个发现也暗示了边界：如果任务分布发生偏移（比如从 web-search 切换到从未见过的 tool-use 场景），那些 behavior-correcting PFs 的 internalized 知识可能仍然有效，但 input-dependent PFs 必须重新构建。PF 的价值可能在于把"可泛化的习惯"和"不可泛化的具体知识"做了清晰分离。

### Q3: Skill Library 的进化如果去掉过滤，为什么会从 60.3% 暴跌到 36.3%？

Table 5 中这个 -24.0 的跌幅非常刺眼。作者解释为"unfiltered evolution introduces library pollution"——垃圾 PF 混入后降低了检索精度和干预质量。

但这背后有个更深层的问题：**PF 的数量和质量之间存在非线性拐点**。少量高质量 PFs 能精准触发并修复问题；但一旦库中混入大量低质量、窄适用的 PFs，检索阶段会召回不相关的 PF，而多个 PF 同时激活时的冲突 resolution 机制并不完善（论文只提到 teacher 辅助选择，未深入讨论 PF 冲突消解策略）。

从系统设计的角度看，这暴露了 HASP 的一个隐含前提：**failure patterns 必须是稀疏且结构化的**——如果失败模式无穷无尽、互不重叠，那么任何技能库都会膨胀到无法维护。HASP 目前只在 web-search、math、coding 三个领域验证了可行性，这些领域的 failure patterns 相对有限且可模式化。更开放的领域（如开放式对话、创意写作），PF 方法是否会遭遇"模式爆炸"？

## 结果速览

| 指标 | 数字 | 一句话解读 |
|---|---|---|
| Inference-time PF-only vs ReAct | **51.0% vs 31.2%** (+25%) | 仅靠执行层干预，零训练就追平或超越多轮 agent |
| + Teacher 选择后 | **56.2%** | teacher 的 PF dispatch 精修带来额外 5 个点 |
| Post-training (HASP-Evolve + RS) | **60.3%** web-search | 超过 Search-R1 (+30.4%) 和 AgentFlow (+7.1%) |
| Math reasoning (Evolve + RS) | **45.4%** | 与 specialized reasoning 方法竞争 |
| Coding pass@1 (w. Teacher) | **68.7%** | 推理时干预接近 SOTA |
|| Coding pass@1 (Evolve + RS) | **69.9%** | 与 GRPO（Group Relative Policy Optimization，基于组内奖励对比的强化学习策略优化）(68.4%) 和 KodCode-RL (66.8%) 竞争 |
| 无过滤 evolution 跌点 | **60.3% → 36.3%** (-24) | skill library 的过滤机制是生死线，不是可选项 |
| Action override 占比 | **65.1%** | PF 多数是硬拦截，而非软提醒 |

## 余味

作者在结论中点出了一个微妙的张力：PF 能 surface 和 stabilize 模型"本来就会但不 reliably 执行"的策略，但**真正全新的策略仍然需要 RL-style 探索**来发现。

这意味着 HASP 不是 RL（Reinforcement Learning，通过与环境交互优化策略的强化学习方法）的替代，而是 RL 的**预处理器**——先把 agent 从"反复犯低级错误"的泥潭中捞出来，让它站在一个更稳定的基线上，然后再用 RL 去寻找更高天花板。如果反过来：先 RL、后 PF，会不会因为基础行为太混乱导致 PF 的 failure patterns 无法收敛？

另一个未被回答的问题是：当 PF 库积累到上千个时，单次激活计算的 overhead 是否会成为 latency 瓶颈？以及，如果多个 PF 同时触发且给出矛盾的修正建议，系统该听谁的？目前的 teacher 辅助选择只解决了"选哪个 PF"，没解决"PF 之间的优先级和冲突消解"。

更深层的结构性疑虑在于：HASP 的整个框架预设了 failure patterns 是**稀疏且可枚举的**。在 web-search、math、coding 这类有明确 success signal 的领域，这个假设成立；但在开放式对话、创意写作、商业策略等反馈模糊的场景中，failure 的边界本身就不清晰，PF 的 `should_activate` 将失去判别标准。这意味着 HASP 的适用范围受限于**环境反馈的可判定性**——它与 AutoHarness 共享同一个 hard boundary：没有廉价、明确的 ground truth，harness 的收敛和进化都会失效。
