# Code as Agent Harness: Toward Executable, Verifiable, and Stateful Agent Systems

> arXiv:2605.18747v1 | 102 pages | Ning, Tieu, Fu, Wei et al. (UIUC, Meta, Stanford), May 2026

---

## 1. Editorial Thesis

编者提出的不是"代码在 agent 中的用途分类"，而是一个**本体论重定位**：代码不是 agent 的输出物或辅助工具，而是 agent 存在的运行基底（harness）——就像操作系统不是应用程序的"工具"，而是程序得以执行的承载体。这个 reframe 的后果是决定性的：当代码成为 harness，**可执行性（executability）成为验证的自然入口，状态（state）成为可检查而非可幻觉的属性，形式化（formality）从锦上添花变为生存条件**。Taxonomy 的三层层叠（Interface → Mechanism → Scaling）本身就是论证：从"代码能做什么"到"代码如何运转"再到"代码如何被共享与演化"，每一步都在把代码从"被 agent 使用的对象"推向"agent 就是被代码编织的进程"。

## 2. Evolution Path

### Phase 1: 代码作为推理的委托载体（Program-of-Thoughts 时代，2022–2023）

**卡在哪了**：LLM 的隐式推理不可检验、不可调试。Chain-of-Thought 把推理过程外化为自然语言，但自然语言不是可执行规格——模型可以"想得对"但"算得错"，且无法自动校验。

**如何回应**：PoT（Program of Thought，将算术推理委托给 Python 解释器执行）、PAL（Program-Aided Language，生成代码后交运行时求值）将推理的**计算部分**卸载到确定性运行时。核心洞察：如果推理的某一步可以被代码表达，那这一步的结果就是可执行的，也就自动可验证。CoC（Chain of Code，将推理步骤写为可执行伪代码）进一步模糊了"代码"与"推理"的边界——代码不再只是计算器，而是推理脚手架。

**遗留问题**：这只是单步委托。代码执行了，但代码本身的正确性谁来验证？推理的分解（decomposition）仍然在 LLM 内部完成，这一步不可审计。

### Phase 2: 形式化验证进入闭环（Prover 时代，2024–2025）

**卡在哪了**：Phase 1 的代码可以被**执行**，但执行通过≠逻辑正确。数字算对了，但证明的步骤可能有漏洞。更根本地，对于数学推理，"运行正确"这个标准本身就不适用——数学命题需要形式化证明，不是数值验证。

**如何回应**：Lean/Isabelle/Coq 等 proof assistant（形式化证明助手，机器可检验的数学证明系统）成为代码的另一个维度。DeepSeek-Prover-V2、Kimina-Prover 等工作将 LLM 与 prover 闭环耦合：LLM 生成证明代码，prover 检验，失败反馈驱动重新生成。这完成了从"可执行"到"可证明"的跃迁。CodePRM（Code-based Process Reward Model，用代码执行过程做过程级奖励信号）和 ExecVerify 则在非形式化场景下模拟类似闭环：用执行轨迹做逐步验证，而非仅看最终结果。

**遗留问题**：形式化验证的覆盖面极窄——绝大多数 agent 任务没有形式化规格。验证本身变成了瓶颈：谁能写出验证条件？

### Phase 3: 代码从推理走向行动——Policy 即程序（Code-as-Policies 时代，2023–2024）

**卡在哪了**：Phase 1-2 解决了"想得对不对"，但 agent 还要"做得对不对"。机器人控制、API 调用、GUI 操作中，LLM 输出的自然语言指令无法被环境直接执行，需要一个翻译层，而翻译本身是错误源。

**如何回应**：Code-as-Policies（CaP，将控制策略直接写为可执行代码而非自然语言指令）实现了范式转换：策略的载体从自然语言变成代码。Voyager（Minecraft 中的终身学习 agent，自动发现并编码技能为 JavaScript 函数）展示了这种载体的进化性——技能以代码库形式积累，可组合、可调试、可版本化，这是自然语言技能库做不到的。SayCan 和 KnowNo 则从约束端入手：用代码定义合法动作的边界，确保 LLM 的"意愿"不超越机器人的"能力"。

**遗留问题**：单 agent 的技能库是孤独的。技能组合的语义冲突、长期运行中的状态漂移、技能复用的泛化边界——这些问题在单 agent 框架内无法被充分提出。

### Phase 4: 代码成为 agent 基础设施——Harness 即系统（2024–至今）

**卡在哪了**：当 agent 从"单任务执行者"变成"持续运行的自主系统"，代码的角色必须从"输出物"升级为"运行时基础设施"。Planning 需要结构化的计划表示而非自然语言流水账；Memory 需要可索引、可压缩的持久化存储而非上下文窗口；Tool Use 需要类型化的接口契约而非自由文本调用；Control 需要沙箱和断言而非模型自律。

**如何回应**：这正是 Survey §3 的核心主张——Planning 用代码表达计划使得计划可验证、可回溯（planning as contract formation）；Memory 用代码结构实现语义索引和上下文压缩；Tool Use 用函数签名和类型系统替代自然语言 API 描述；Control 用沙箱执行和确定性传感器（deterministic sensors，不依赖 LLM 判断的硬检验机制）替代模型自我约束。§3.5 Agentic Harness Engineering 进一步提出：harness 本身应被 telemetry（深遥测，全链路运行时监控）覆盖，并允许 evolution agent（演化智能体，负责修改 harness 自身的 agent）在 governed mutation（受治理的变异，变更须通过验证才能提交）下进化。

**遗留问题**：所有这些机制都在单 agent 假设下设计。当多个 agent 共享代码基底时，语义冲突、状态一致性、变异的交叉影响——这些分布式系统的经典问题在 agent harness 层面完全开放。

### Phase 5: 多 Agent 共享代码基底——Harness 即协议（前沿）

**卡在哪了**：Phase 4 的 harness 是私有的。每个 agent 有自己的 plan/memory/tool/control，彼此只能通过自然语言通信。这在单 agent 场景下可行，但多 agent 协作中，自然语言通信的歧义和不可验证性成为协作的天花板。

**如何回应**：§4 提出以代码为共享基底（shared code-centric harness substrate）。角色的专业化（manager/planner/coder/reviewer/tester）不再只是分工，而是**代码所有权的划分**；交互模式（programming/repair/debate/red-teaming）本质上是**代码变更的协商协议**；共享表示（shared representation）和 harness-state convergence（harness 状态收敛，多 agent 的 harness 状态向一致方向演化）试图解决分布式 agent 的一致性问题——这本质上是在 agent 层面重演分布式系统从"消息传递"到"共享状态"的演化。

**遗留问题**：代码共享带来的语义冲突远比数据共享复杂。两个 agent 对同一函数的不同修改意图，在没有事务语义（transactional semantics，保证操作原子性和一致性的机制）的情况下，可能产生语义上合法但逻辑上矛盾的代码。这正是 §5.2 Open Problem 4 的核心。

## 3. Structural Map

```
§2 Harness Interface（代码能表达什么）
  ├── Reasoning（推理的可执行化）── 确定性作为验证入口
  ├── Acting（策略的程序化）────── 可组合性作为能力积累入口
  └── Environment（世界的结构化）── 可模拟性作为安全测试入口
          │
          ▼ 因果依赖：Interface 决定了 Mechanism 可以操作什么
§3 Harness Mechanisms（代码如何运转）
  ├── Planning ──── 将 Interface 中的 Reasoning 结构化为可执行计划
  ├── Memory ────── 将 Interface 中的 Acting 经验持久化为可检索代码
  ├── Tool Use ──── 将 Interface 中的 Environment 接口化为类型化函数
  ├── Control ───── 在 Mechanisms 之上叠加安全边界
  └── Engineering ── 将全部 Mechanisms 纳入可观测、可变异的治理框架
          │
          ▼ 因果依赖：Mechanism 的复杂性超出单 agent 管理能力
§4 Scaling the Harness（代码如何被共享）
  ├── Role Specialization ── 对 Mechanisms 的所有权划分
  ├── Interaction Modes ──── 对代码变更的协商协议
  ├── Workflow Topology ──── 对执行依赖的调度结构
  └── Shared Substrate ──── 对一致性的最终保障
          │
          ▼ 因果依赖：Scaling 暴露的裂缝构成 Open Problems
§5 Open Problems
  └── 每个问题都是前一层未解决的张力的显化
```

依赖链：Interface 定义可操作对象 → Mechanism 定义操作方式 → Scaling 暴露操作冲突 → Open Problems 承认冲突的不可解性（当前）。

## 4. Mechanism Deep-Dive

### 张力 A：可执行性 vs. 语义正确性

贯穿 Phase 1 到 Phase 5。代码可执行意味着运行时不会崩溃，但不意味着逻辑正确。PoT 解决了"算术推理的数值正确性"，Lean prover 解决了"数学证明的逻辑正确性"，但 agent 的大多数任务处于两者之间——既没有形式化规格，也不能仅靠执行通过来判断。CodePRM 和 ExecVerify 试图用执行轨迹做过程级验证，但这本质上是**用可观测信号近似不可观测的语义正确性**，是一个无法闭合的近似。§5.2 Open Problem 2（Semantic Verification Beyond Executable Feedback）正是这个张力的最新显化：当执行反馈不足以验证语义时，我们是否需要一种全新的验证范式，还是承认语义正确性在 open-ended 任务中不可自动化验证？

### 张力 B：Harnessed Autonomy——约束力 vs. 表达自由度

从 SayCan 的 grounded skill selection 到 §3.4 的 sandboxed execution，harness 的核心承诺是"通过约束保证安全"，但约束必然压缩策略空间。SayCan 选择只约束动作合法性（最弱约束），Voyager 选择让 agent 自由探索但用代码库做能力积累（中等约束），§3.5 的 governed mutation 选择允许 harness 变异但要求验证通过（有条件自由）。这个张力的深层结构是：**harness 越强，agent 越安全但也越笨；harness 越弱，agent 越灵活但也越危险**。§5.2 Open Problem 5（Human-in-the-Loop Safety and Accountability）试图引入人类判断作为弹性安全网，但这只是把张力从"代码 vs. agent"转移到"人类 vs. 系统"——人类审阅者能理解和审计的代码复杂度有上限，而 harness 的复杂度没有上限。

### 张力 C：代码共享的一致性代价

从 Phase 4 到 Phase 5 的跃迁中浮现。单 agent 的 harness 是私有状态，不存在一致性问题。多 agent 共享代码基底时，每个 agent 对共享代码的理解和修改意图可能冲突。§4 提出的 harness-state convergence 是一个目标而非机制——它描述了"状态应该收敛"但未说明"如何保证收敛"。这直接映射到分布式系统中的共识问题，但比传统共识更难：传统共识解决的是"对同一数据值达成一致"，而 agent harness 共享需要解决的是"对代码的语义达成一致"——同样的代码文本，不同 agent 可能理解为不同的行为规格。§5.2 Open Problem 4（Transactional Shared Program State and Semantic Conflict Resolution）是这个张力的显式承认，但提出的"transactional"隐喻可能低估了问题的难度——数据库事务解决的是写冲突，而代码共享的冲突是语义冲突，不是写冲突。

## 5. Open Problems & Frontiers

### 编者真在意的问题（核心张力驱动的必然产物）

**P1. Harness-Level Evaluation and Oracle Adequacy**：当 harness 成为 agent 的运行基底，"agent 好不好"的评价就分解为"harness 好不好"和"agent 在 harness 内好不好"。但评价 harness 需要什么 oracle？如果用 LLM 评价 LLM 的 harness，评价者本身不可信；如果用形式化验证，大多数任务没有规格；如果用环境反馈，反馈信号的覆盖度有限。这不是方法选择问题，而是**评价的锚点缺失**——我们缺乏一个不依赖于被评价系统的 ground truth 来源。编者将此列在 §5.2 首位，暗示这可能是整个框架最脆弱的地基。

**P3. Self-Evolving Harnesses without Regression**：§3.5 的 governed mutation 允许 harness 变异，但"通过验证才提交"假设了验证条件的完备性。如果验证条件本身有盲区，harness 的进化可能在"看起来正确"的方向上持续退化。这在软件工程中是已知问题（测试覆盖度≠正确性覆盖度），但在 agent harness 中更危险——harness 的退化可能表现为 agent 行为的微妙偏移，而非显式崩溃，从而更难被检测。编者将其列为独立问题而非 governed mutation 的子问题，暗示其认为**当前的验证范式从根本上不足以支撑 harness 的无回归进化**。

**P4. Transactional Shared Program State and Semantic Conflict Resolution**：如张力 C 所述，这是多 agent harness 的核心难题。编者用了"transactional"这个隐喻，但紧接着提到"semantic conflict resolution"——后者才是真正的挑战。数据库事务的前提是操作语义清晰（读写原子值），代码修改的语义是上下文相关的（同一段代码在不同调用路径下意义不同）。编者似乎意识到隐喻的局限，但未提出替代框架。这个问题可能需要从程序分析（program analysis，自动推导代码行为的技术）而非数据库领域寻找工具。

**P7. Toward a Science of Harness Engineering**：编者将此放在最后，但这是对整个 survey 的元反思——如果我们有了可执行的推理、可验证的行动、可结构化的环境、可治理的机制、可共享的基底，我们仍然缺乏一个**预测性理论**来回答"harness 的这个修改会导致 agent 行为如何变化"。当前所有 harness 工程都是经验性的：改了、跑了、看了。编者隐含的期望是，harness 工程应该从手艺走向科学——但 102 页的 survey 本身就是"这个科学还不存在"的证据。

### 礼貌性展望（有方向但张力不足）

**P2. Semantic Verification Beyond Executable Feedback**：重要但不紧急。形式化验证社区已深耕数十年，agent 场景的特殊性（开放世界、模糊规格）未必催生新方法，更可能是现有方法的不充分应用。编者将其列在第二位可能是因为 survey 的叙事完整性需求，而非实质性紧迫性。

**P5. Human-in-the-Loop Safety and Accountability**：这是任何自主系统的标准免责声明。编者未提出超出已有 HITL（Human-in-the-Loop，人类参与决策循环的安全模式）框架的新洞察——"人类应该有权审计和干预"是共识而非问题，真正的问题（人类审计能力的上限、责任归属的法律框架）被搁置。

**P6. Multimodal Code-Harness Systems**：视觉、语音等模态如何融入代码 harness 是一个工程扩展方向，但缺乏概念性挑战。当前代码 harness 的核心优势（可执行、可验证、有状态）在多模态下是否仍然成立？编者未讨论这个前提，而是默认成立后讨论"如何融入"，这更像路线图而非研究问题。

## 6. Critical Audit

### Taxonomy 的隐含假设：代码的统一性

Survey 的核心修辞是将 Reasoning/Acting/Environment 统一在"代码"之下，暗示代码是一种足够通用的表示来承载 agent 的全部运行时需求。但这忽略了代码内部的**异质性断裂**：推理代码（Python 算术表达式）是确定性的、可逆的；策略代码（机器人控制程序）是时序敏感的、不可逆的；环境建模代码（物理模拟器）是近似性的、有精度限制的。这三类代码对验证的要求截然不同——推理代码验证逻辑正确性，策略代码验证时序安全性，环境代码验证保真度——但 Survey 将它们平铺在同一层（§2），未讨论这种异质性对 harness 统一性的挑战。

### "Harness"隐喻的边界

Harness 的原意是马具——约束马的力量使其为人所用。这个隐喻暗示了一个被约束的主体（agent）和一个约束的工具（harness），且两者可分离。但在 §3.5 和 §4 中，harness 本身被 agent 修改、被多 agent 共享、参与自身演化——harness 与 agent 的边界正在模糊。当 harness 可以修改自身时，它不再是"工具"而是"参与者"；当多 agent 共享 harness 时，harness 不再属于任何一个 agent 而是属于群体。Taxonomy 仍然假设 harness 是 agent 的外部附属物，但 survey 自己的内容已经在消解这个假设。这不是错误，而是一个值得显式承认的张力——**harness 的自治性（autonomy）是 survey 最大的未言明前提**。

### §3 的平铺倾向

§3 将 Planning/Memory/Tool Use/Control/Engineering 并列，但这种平铺掩盖了它们之间的因果层级。Planning 产生计划，计划需要 Memory 存储，Memory 中的经验指导 Tool Use 的选择，Tool Use 的结果需要 Control 约束，全部需要 Engineering 治理——这是一条依赖链而非并列关系。平铺处理导致 §3 读起来像系统设计文档而非论证，失去了 §2 的论证力度。

### 漏洞：对抗性环境的缺位

整篇 survey 假设环境是协作性的或至少是中性的——环境不会主动欺骗 agent。但对抗性环境（adversarial environment，故意提供虚假反馈的部署环境）下，harness 的核心优势变为核心弱点：可执行性意味着恶意环境可以诱导代码执行有害操作；可验证性意味着验证信号本身可能被污染；有状态意味着状态可能被污染。§3.4 的 sandboxed execution 部分缓解了执行风险，但未讨论验证信号的可靠性。这是 survey 最大的盲区。

## 7. Brainstorm Q&A

### 对齐入门

**Q1: "Code as Harness"和"Code as Output"的根本区别是什么？**
→ Code as Output：代码是 agent 的产出物，一次性生成后交付，agent 与代码的生命周期解耦。Code as Harness：代码是 agent 的运行时基底，agent 持续驻留在代码中，代码的状态就是 agent 的状态。锚点：Voyager 的技能代码库——agent 不只是"写出"技能函数，而是"活在"技能函数构成的能力空间中。

**Q2: 为什么 "可执行"比"自然语言"更适合做 agent 的运行基底？**
→ 三个原因叠加：(1) 可执行意味着结果可自动校验，无需人类审阅；(2) 可执行意味着状态可观测——变量的值是确定的而非模糊的；(3) 可执行意味着组合可预测——函数调用的结果可推导。自然语言在这三个维度上都是模糊的。锚点：PoT 将"2+3=?"从 LLM 的隐式计算变为 Python 的 `2+3`，结果从"可能对"变为"必然对"。

**Q3: Harness 和传统软件架构中的 middleware 有什么区别？**
→ Middleware 是请求-响应管道中的横切关注点处理层（日志、认证、缓存），它不改变业务逻辑。Harness 则直接参与 agent 的推理和行动——它不只是传递请求，而是定义 agent 可以做什么、如何做、做到什么程度。锚点：§3.4 的 planning as contract formation——harness 不只是执行计划，而是约束计划本身的合法性。

### 张力辨析

**Q4: 如果 harness 可以自我修改（§3.5 governed mutation），如何区分"进化"和"退化"？**
→ 当前答案是"通过验证条件的检验"，但这假设验证条件是完备的——而验证条件本身也是代码，也可能有 bug。这导致递归问题：谁验证验证条件？形式化方法可以打断递归（用 prover 验证验证器），但形式化规格的编写又需要人类。最终，harness 进化的可信赖性可能需要锚定在人类审计上——但 §5.2 P5 暗示人类审计能力有上限。锚点：governed mutation 的 governance 本身是未验证的假设。

**Q5: 多 agent 共享 harness 时，"代码即规格"的假设是否仍然成立？**
→ 单 agent 场景下，agent 自己写自己读的代码是自洽的——写者即读者，理解一致。多 agent 场景下，写者和读者分离，同一代码文本可能被不同 agent 解读为不同行为——"代码即规格"退化为"代码即文本"，规格的确定性来自解读的一致性而非文本本身。锚点：§4 的 harness-state convergence 是目标描述而非机制描述。

**Q6: 张力 A（可执行性 vs. 语义正确性）是否存在非近似的解决方案？**
→ 理论上，如果所有任务都有形式化规格，形式化验证可以闭合间隙。但形式化规格的编写成本和开放世界任务的规格不完备性使得这不可能。另一种思路：放弃"自动化验证语义正确性"的目标，转而追求"语义错误可检测"——不保证正确，但保证错误不会无声发生。这类似软件工程中的 defensive programming（防御性编程，在代码中嵌入运行时检查），但在 harness 层面可能表现为断言注入和运行时监控。锚点：§3.4 的 deterministic sensors 是这个方向的雏形。

### 盲区推演

**Q7: 如果环境是主动对抗的，harness 的哪些优势会变成劣势？**
→ 可执行性 → 恶意环境可诱导代码执行非预期操作（如注入恶意 API 响应）；可验证性 → 验证信号被污染，agent 基于虚假反馈修改 harness；有状态 → 状态被污染后，后续所有决策基于错误前提。最危险的场景：对抗性环境提供看似合理但实际有毒的反馈，诱导 governed mutation 将有害修改写入 harness。锚点：survey 全文未讨论对抗性环境。

**Q8: 编者将 "Science of Harness Engineering"（P7）列为 open problem，但未提出任何候选理论。如果必须提出一个，最可能的方向是什么？**
→ 候选：基于程序分析（program analysis）的 harness 行为预测理论。给定 harness 代码和 agent 行为规格，自动推导 agent 在 harness 约束下的可达行为空间。这需要将抽象解释（abstract interpretation，在不执行程序的情况下推导程序行为近似的技术）适配到 agent-harness 交互场景。但核心困难是：agent 的行为不可完全预测（LLM 是黑盒），harness 的约束效果只能被近似分析。锚点：§3.4 的 debugging → harness-level control 已经在做局部的行为约束分析，但缺乏全局理论。

**Q9: survey 将 "code"视为统一载体，但如果未来多模态 agent 的主要交互方式是视觉-动作而非文本-代码，harness 框架是否仍有解释力？**
→ 解释力会衰减。代码 harness 的优势来自文本表示的可组合性、可验证性、可审计性。视觉-动作的主要载体是感知-运动轨迹（perceptual-motor trajectory），这些轨迹不自然地映射到代码表示。§5.2 P6（Multimodal Code-Harness）假设多模态可以"融入"代码 harness，但更可能的情况是：多模态 agent 需要 harness 的概念但不需要代码的载体——harness 的功能（约束、验证、状态管理）需要被重新实现在非代码的表示上。锚点：SkillVLA（§2.2）已经展示了视觉-语言-动作技能的编码，但其"代码"部分主要是调度逻辑而非感知-动作本身。
